## Tech Stack Rules

### Backend
- Framework: **Hono.js** (NOT Express, NOT NestJS, NOT Fastify)
- ORM: **Prisma** (NOT raw SQL, NOT TypeORM, NOT Drizzle)
- Language: **TypeScript** with strict mode
- Pattern: Route ŌåÆ Controller ŌåÆ Service ŌåÆ Repository (always, no shortcuts)
- Auth: Validate AFHome JWT token ŌĆö never create a separate auth system

### Frontend
- Framework: **Next.js App Router** (NOT Pages Router)
- UI: **shadcn/ui** components (NOT custom CSS components from scratch)
- Styling: **Tailwind CSS** only (NO CSS modules, NO styled-components)
- State (server): **RTK Query** (NOT useEffect + fetch, NOT React Query, NOT SWR)
- State (UI): **Redux Toolkit** (NOT Zustand, NOT Context API for global UI state)

## Code Generation Rules

### DO
- Follow the Route ŌåÆ Controller ŌåÆ Service ŌåÆ Repository pattern strictly
- Use shadcn/ui components ŌĆö import from `@/components/ui/`
- Use `cn()` from `@/lib/utils` for conditional Tailwind classes
- Use RTK Query for all API calls in the frontend
- Use Prisma for all database access in the backend
- Use AFHome JWT token in `Authorization: Bearer <token>` header for all community API requests
- Export Prisma client as a singleton from `src/lib/prisma.ts`
- Keep Hono route files thin ŌĆö just register routes and middlewares

### DO NOT
- Do not create a separate login/signup system ŌĆö auth comes from AFHome
- Do not put Prisma queries in controllers or route handlers
- Do not use `useEffect` for data fetching in React components
- Do not use inline styles or CSS modules
- Do not use `any` TypeScript type
- Do not add `console.log` statements ŌĆö use a proper logger
- Do not hardcode API URLs ŌĆö use environment variables
- Do not use Pages Router (`pages/` directory) ŌĆö use App Router (`app/` directory)
- Do not create new shadcn components from scratch if an existing one fits

## Prisma Rules

- Always generate client to `src/generated/prisma` (as configured in schema)
- Use `cuid()` for all primary keys (not `uuid()`, not auto-increment)
- Use `snake_case` for all database field names in `@map()` if needed
- Always include `createdAt` and `updatedAt` on every model
- Run `prisma generate` after schema changes
- Run `prisma migrate dev` for local schema migrations

## API Response Rules

All API responses must follow this format:

**Success:**
```json
{ "data": { ... } }
{ "data": [...], "meta": { "total": 100, "page": 1, "limit": 20 } }
```

**Error:**
```json
{ "error": "Human-readable message", "code": "SNAKE_CASE_CODE", "status": 404 }
```

HTTP status codes:
- 200: Success (GET, PUT)
- 201: Created (POST)
- 204: No Content (DELETE)
- 400: Bad Request (validation error)
- 401: Unauthorized (missing/invalid token)
- 403: Forbidden (valid token, insufficient permissions)
- 404: Not Found
- 500: Internal Server Error

## Auth Rules

- Community backend validates AFHome tokens on every protected route
- Use `authMiddleware` from `src/middleware/auth.ts` ŌĆö apply at router level, not per-route
- The validated user object is stored in Hono context: `c.get('user')`
- User identity fields available: `id`, `name`, `email`, `avatar`, `tier`
- Never trust client-provided `authorId` ŌĆö always use `c.get('user').id`

## UI Generation Rules

When building UI components, the reference design is the Neighborly/Nextdoor screenshot:
- 3-column layout: sidebar | feed | right panel
- White cards on gray background
- Blue primary color for CTAs and active states
- Rounded corners on all cards (`rounded-xl`)
- Subtle shadows (`shadow-sm`)
- Gray muted text for timestamps and metadata

When in doubt, refer to `6_file_context/ui-context.md.txt` for the exact component breakdown.
